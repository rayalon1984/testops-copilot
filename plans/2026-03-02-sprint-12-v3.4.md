## Part 1 — PRODUCT: Feature Plan

> Persona: `AI_PRODUCT_MANAGER`

### Problem Statement

Sprint 11 (v3.3.0) shipped the Xray Cloud integration foundation: manual push sync, settings UI, `xray_search` AI tool, and connection validation. However, feedback from the sprint review reveals three gaps:

1. **Manual sync friction** — QA engineers must remember to click "Sync to Xray" after every test run. Teams with 20+ runs/day find this unsustainable. The auto-sync toggle was a P1 that didn't land.
2. **No test plan visibility** — Engineering leads can't see Xray test plan coverage percentages from Copilot. They still context-switch to Jira/Xray to check plan health.
3. **AI analysis is Xray-blind** — When the copilot analyzes a failure, it doesn't know that the failing test is mapped to Xray test case `PROJ-TC-102` with 3 prior failures. This context would dramatically improve root cause analysis accuracy.

Additionally, the V2 copilot cards have been validated behind a feature flag for two sprints. Keeping the V1 code path increases maintenance burden (dual rendering logic, double the test surface). It's time to graduate V2 to default and remove dead V1 code.

Finally, two feature specs (`xray-integration`, `smart-starter-prompts`) remain in `draft` status despite being shipped. Promoting them to `active` with full assertion coverage closes a process gap.

## Part 2 — ARCHITECT: Architectural Design

> Persona: `AI_ARCHITECT`

### Architecture Decision 1: Event-Driven Auto-Sync

**Decision**: Implement auto-sync as a service-level event listener on test run completion, not as a cron job or polling mechanism.

**Rationale**:
- The `TestRunService` already transitions runs to COMPLETED status — adding an event hook there keeps the trigger colocated with the state change
- Event-driven eliminates polling latency and wasted cycles (cron would check all runs every N seconds)
- Aligns with the existing pattern: notifications already fire on test run completion via `NotificationService`
- The auto-sync check is a simple guard: `if (config.xray?.autoSync && xrayService.isEnabled())`

**Rejected alternatives**:
- Cron-based polling — wasteful for low-volume orgs, introduces latency for high-volume ones
- Queue-based (Redis/Bull) — over-engineering for v3.4; viable for v3.5 if sync volume grows

**Implementation**: Add a post-completion hook in `TestRunService.completeRun()`:

```typescript
// In test-run.service.ts — after status transitions to COMPLETED
if (config.xray?.autoSync && xrayService.isEnabled()) {
  // Fire-and-forget — don't block the response
  xrayService.syncTestRun(testRun.id).catch((err) => {
    logger.warn('Auto-sync to Xray failed', { testRunId: testRun.id, error: err.message });
  });
}
```

**Key design choice**: Fire-and-forget (non-blocking). The test run completion response should not wait for Xray sync. Failures are logged and visible in sync history, but never block the user.

### Architecture Decision 2: Test Plan Browser — Thin Proxy Pattern

**Decision**: The test plan browser is a thin proxy to the Xray Cloud GraphQL API. Copilot does NOT replicate or cache test plan data.

**Rationale**:
- Xray is the source of truth for test plans and coverage
- Caching plan data would require invalidation logic (plans change frequently)
- The Xray API supports pagination and filtering natively — no need to re-implement
- Consistent with the v3.3 design: `XraySync` tracks sync state only, not Xray data

**Xray GraphQL query for test plans**:
```graphql
query GetTestPlans($projectId: String!, $limit: Int, $start: Int) {
  getTestPlans(projectId: $projectId, limit: $limit, start: $start) {
    total
    results {
      issueId
      jira(fields: ["summary", "key"])
      testCoverage { covered, total, percentage }
      lastModified
    }
  }
}
```

### Architecture Decision 3: AI Enrichment with Xray Context

**Decision**: Extend the existing `enrichment.service.ts` pipeline to include an Xray enrichment step, parallel to Jira/Confluence/GitHub enrichment.

**Rationale**:
- The `POST /api/ai/enrich` endpoint already runs parallel enrichment queries (Jira, Confluence, GitHub)
- Adding Xray as a fourth parallel source fits the existing pattern perfectly
- The enrichment result is added to the AI system prompt as structured context
- Xray enrichment is gated on: (1) Xray is configured, (2) the test result has `externalTestCaseId`

**Enrichment data shape**:
```typescript
interface XrayEnrichmentContext {
  testCaseKey: string;           // e.g., "PROJ-TC-102"
  summary: string;               // Test case title
  status: string;                // Current status (PASS/FAIL/TODO)
  executionHistory: Array<{      // Last 5 executions
    date: string;
    status: string;
    executionKey: string;
  }>;
  linkedDefects: Array<{         // Linked Jira defects
    key: string;
    summary: string;
    status: string;
  }>;
}
```
